## 📝 1. 실시간 이상거래 탐지(FDS) 시스템 요구사항 정의서## 1. 개요

* 프로젝트명: Apache Kafka & Drools CEP 연동 기반 실시간 이상거래 탐지(FDS) 시뮬레이터
* 목적: 고성능·저지연 실시간 데이터 스트림 파이프라인 아키텍처를 가상화하여 이상 결제 징후를 실시간 평가하고 전송 제어할 수 있는 대시보드 구축

## 2. 시스템 아키텍처 및 핵심 요구사항 (3대 포인트)

   1. 이벤트 시간(Event Time) 중심 평가: 네트워크 지연 왜곡을 방지하기 위해 서버 시스템 시간이 아닌 메시지 고유 타임스탬프 기반의 의사 시계(PseudoClock)를 활용해 규칙을 평가함.
   2. 컨슈머 스케일 아웃 및 분산 메모리: 동일 사용자(userId)의 과거 이벤트를 한곳에서 연속 평가할 수 있도록 사용자 ID를 Kafka 파티션 키로 지정하여 논리적 샤딩(Sharding) 처리를 보장함.
   3. 규칙(Rule)의 동적 변경: 애플리케이션의 재배포 및 중단 없이, 런타임 환경에서 슬라이딩 윈도우 크기와 탐지 임계치를 동적으로 반영함.

## 3. 세부 기능 요구사항 (F-REQ)## F-REQ-01: 데이터 파이프라인 시뮬레이션 제어 (Streamlit UI)

* F-REQ-01-1: 사용자는 대시보드 화면에서 "가상데이터 발생 ⚡" 버튼을 클릭하여 시뮬레이션을 시작할 수 있다.
* F-REQ-01-2: 시뮬레이션 시작 시, 시스템은 0.5초 주기로 무작위 금융 거래 이벤트를 연속 생성하여 파이프라인에 주입한다.
* F-REQ-01-3: 사용자는 대시보드 화면에서 "가상데이터 발생 중지 🛑" 버튼을 클릭하여 데이터 발생 및 파이프라인 스트리밍을 즉시 중단(동결)할 수 있다.

## F-REQ-02: 가상 Kafka 프로듀서 및 토픽 분리

* F-REQ-02-1: 수집된 가상 이벤트는 payment-events 토픽으로 발행된다.
* F-REQ-02-2: 이벤트 생성 시 사용자 ID를 파티션 키로 매핑하여 특정 컨슈머 인스턴스로 전달되는 분산 환경을 모사한다.
* F-REQ-02-3: 이상 거래 감지 시 즉각 경고 메시지를 fds-alerts 토픽으로 발행하고 결제 즉시 차단 API 호스팅 상태로 연동한다.

## F-REQ-03: Drools Fusion(CEP) 모사 엔진 및 인메모리 관리

* F-REQ-03-1: 사용자 ID 기반으로 격리된 세션 메모리(Stateful Session)를 램(RAM) 상에 유지한다.
* F-REQ-03-2: Sliding Window 규칙을 통해 설정된 탐지 범위 시간이 지난 과거 데이터는 인메모리에서 자동으로 만료(Evict) 처리한다.
* F-REQ-03-3: [타국 결제 규칙] 슬라이딩 윈도우 시간 내에 한 사용자가 2개국 이상의 국가(예: KR, US, VN)에서 결제를 시도하면 실시간으로 이상거래 경고를 트리거한다.
* F-REQ-03-4: [과다 결제 규칙] 슬라이딩 윈도우 시간 내에 설정된 임계 횟수 이상의 연속 결제가 인입될 경우 위험 징후로 탐지한다.

## 4. 기술 및 환경 요구사항 (N-REQ)

* N-REQ-01: 프론트엔드 UI 화면과 핵심 비즈니스 로직(함수 및 클래스)은 파일 레벨에서 철저히 격리되어야 한다 (util/ 디렉토리 하위로 분리).
* N-REQ-02: Streamlit 버전 업데이트 정책에 따라 감쇠 경고를 유발하는 구형 파라미터(use_container_width) 사용을 금지하며, 최신 표준 규격인 width='stretch'를 채택해야 한다.
